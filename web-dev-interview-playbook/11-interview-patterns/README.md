## Why Patterns Matter

Interviewers rarely ask truly novel questions. They ask the same **shapes** in different words:

| You hear | It's really this pattern |
|----------|--------------------------|
| "Why React and not Vue?" | Why-question |
| "REST vs GraphQL?" | Comparison |
| "Login works locally but not in prod" | Debugging |
| "How would you scale this?" | Architecture |
| "Is that the best database for this?" | Project defense |

If you prepare by **pattern**, one good answer framework covers dozens of phrasings.

## How to Study These

### Step 1: Read the answer out loud (don't skim)

Silent reading feels productive. It's not interview practice.

### Step 2: Replace the example with YOUR project

Every answer has a project connection section. Fill it in:

> "Why MongoDB?" → "In my **expense tracker**, I chose MongoDB because…"

### Step 3: Practice the follow-ups

The main answer gets you 60 seconds. Follow-ups are where interviews are won or lost.

Pair with [answer-frameworks.md](../00-start-here/answer-frameworks.md):

## Common Mistakes Across All Patterns

1. **No tradeoff** — "X is the best" sounds junior. "X fits because…, but Y is better when…" sounds thoughtful.
2. **Tutorial answer** — "I used it because the tutorial used it" is honest but weak. Add one technical reason.
3. **No project link** — Always bridge back: "In my project, this mattered because…"
4. **Memorized essay** — Keep answers under 90 seconds unless they ask for more.
5. **Fake experience** — If you haven't used X, say so and explain how you'd evaluate it.

## Pair With Mock Interviews

After studying patterns, run a mock where your partner **only** asks why/comparison/debugging questions for 20 minutes. It's harder than it sounds.
